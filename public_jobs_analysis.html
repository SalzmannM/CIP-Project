1 Introduction

This project analyzes job advertisements within the Swiss public sector to examine how job requirements and working conditions vary across regions, occupations, and organisation types (cantons, municipalities, hospitals, etc.). It demonstrates how web-scraped data can be gathered, cleaned, and analysed within a complete data-science workflow to answer concrete research questions through visualisation and statistical analysis. The primary data source is publicjobs.ch, a dedicated job platform for the Swiss public sector, scraped via Selenium and BeautifulSoup. To contextualise the observed workload distribution against the broader Swiss labour market, the dataset is enriched with official employment statistics from the Swiss Federal Statistical Office (BFS), specifically the Beschäftigungsstatistik (BESTA) retrieved via the PX-Web API.

The analysis is guided by three research questions: (RQ1) How does Swiss public-sector labour demand distribute geographically and by occupational class? (RQ2) Are job postings statistically distinguishable from the BFS workforce benchmark in terms of full-time / part-time split? (RQ3) Which fringe benefits are most commonly advertised, and how do they vary between part-time and full-time roles and across cantons?

2 Data Acquisition

2.1 Legal, Ethics, and Access

All data used in this project come from publicly accessible sources. The robots.txt file of publicjobs.ch was reviewed prior to data collection to ensure compliance with its crawling policies. We strictly avoided restricted paths (e.g., direct application links). The platform’s robots.txt permits general access to /jobs listing pages, which is the only area our scraper interacts with.

To extract the data, we implemented a dynamic web scraping approach using Selenium. This was necessary to properly render JavaScript-loaded elements, handle cookie banners, and interact with the dynamic pagination. We utilized polite pauses (randomized time intervals) between page requests to minimize server load and emulate natural browsing behavior. Additional data from the BFS are used under open data terms, and no personal or user-related information is collected.

2.3 Dynamic Data Extraction with Selenium

The following script initializes a Chrome WebDriver, navigates the job platform, explicitly declines tracking cookies via DOM interaction, and systematically extracts structured information from the job cards. publicjobs.ch renders its listings dynamically via JavaScript, requiring Selenium to interact with the live DOM rather than parsing static HTML alone.

The scraper performs the following steps:

  1. A Chrome WebDriver instance is started and navigated to the /jobs overview page. After the initial page load, a cookie consent banner is dismissed via an explicit DOM click on the decline button.
  2. On each page, BeautifulSoup parses the rendered HTML to locate all job card elements (div.molecule-business-advertisement). For each card, the scraper extracts the job title and detail URL from the first anchor tag, the company name from a labeled span element, and structured metadata fields (posting date, occupation field, workload, city, organization type) by matching Font Awesome icon classes to their corresponding value spans.
  3. The raw workload string (e.g., “80 - 100”, “Vollzeit”, “60”) is parsed into normalized FTE boundaries (fte_from, fte_to) and a categorical work-time label. The canton is inferred from the URL path structure.
  4. After all cards on a page are processed, the scraper clicks the “Nächste Seite” pagination button using JavaScript execution (to avoid overlay-related click interception) and waits a randomized interval before loading the next page. A duplicate check based on detail URLs prevents repeated entries.

# --- Scraper Configuration ---
driver = webdriver.Chrome()
base_url = "https://publicjobs.ch"

# How many pages to scrape (73 pages available as of March 2026)
max_pages = 73

# Start on the jobs overview page
driver.get(f"{base_url}/jobs")

# Wait for initial page load
time.sleep(5)

# --- Close Cookie Banner ---
try:
    # Wait up to 5 seconds until cookie-banner decline button is clickable
    decline_button = WebDriverWait(driver, 5).until(
        EC.element_to_be_clickable((By.ID, "hs-eu-decline-button"))
    )
    decline_button.click()
    print("cookie-banner declined")
    
    # Short break until cookie-banner animation completes
    time.sleep(1.5) 
except Exception:
    print("did not find a cookie banner")

# --- Main Scraping Loop ---
jobs = []
seen_urls = set()

for page in range(1, max_pages + 1):

    print(f"\n--- Page {page} ---")

    # Polite pause: wait for jobs to load after page change
    time.sleep(random.uniform(2.5, 4.5))

    html = driver.page_source
    soup = BeautifulSoup(html, "html.parser")

    # Find all job cards on the page
    # Each job is a <div class="molecule-business-advertisement ...">
    articles = soup.find_all("div", class_="molecule-business-advertisement")

    print(f"  Jobs on this page: {len(articles)}")

    for article in articles:
        try:

            # --- Extract detail URL and title ---
            # The first <a> tag contains href and title attributes
            link_tag = article.find("a", href=True)

            if not link_tag:
                print("  No link found → skipping")
                continue

            detail_path = link_tag.get("href", "")
            detail_url = base_url + detail_path
            title = link_tag.get("title", "")

            # Duplicate check via URL set
            if detail_url in seen_urls:
                continue
            seen_urls.add(detail_url)

            if not title:
                # Fallback: extract title from h3 element
                h3 = article.find("h3")
                title = h3.get_text(strip=True) if h3 else None

            # --- Extract company name ---
            # Company is in <span class="...molecule-business-advertisement__owner">
            company_tag = article.find(
                "span", class_=lambda c: c and "molecule-business-advertisement__owner" in c
            )
            company = None
            if company_tag:
                val = company_tag.find("span", class_="molecule-typography-label__value")
                company = val.get_text(strip=True) if val else None

            # --- Extract structured fields via Font Awesome icon classes ---
            # publicjobs uses FA icons to label each metadata field:
            #   fa-calendar     → posting date
            #   fa-briefcase    → occupation field (Berufsfeld)
            #   fa-percent      → workload (Vollzeit / 50-70 / etc.)
            #   fa-location-dot → city
            #   fa-building     → organisation type (Branche)

            posting_date = None
            occupation_field = None
            workload_raw = None
            city = None
            org_type = None

            # Find all label spans that contain an icon
            all_labels = article.find_all(
                "span", 
                class_=lambda c: c and "molecule-typography-label" in c and "non-selectable" in str(c)
            )

            for label in all_labels:
                # Skip the company label (already extracted)
                if "molecule-business-advertisement__owner" in str(label.get("class", [])):
                    continue

                # Find the icon inside this label
                icon = label.find("i", class_=lambda c: c and "atom-icon-fontawesome" in c)
                # Find the value span
                value_span = label.find("span", class_="molecule-typography-label__value")

                if not icon or not value_span:
                    continue

                icon_classes = " ".join(icon.get("class", []))
                value = value_span.get_text(strip=True)

                # Match icon class to field
                if "fa-calendar" in icon_classes:
                    posting_date = value
                elif "fa-briefcase" in icon_classes:
                    occupation_field = value
                elif "fa-percent" in icon_classes:
                    workload_raw = value
                elif "fa-location-dot" in icon_classes:
                    city = value
                elif "fa-building" in icon_classes:
                    org_type = value

            # --- Parse workload string into FTE boundaries ---
            fte_from = None
            fte_to = None
            work_time = None

            if workload_raw:
                if workload_raw.lower() == "vollzeit":
                    work_time = "Vollzeit"
                    fte_from = 1.0
                    fte_to = 1.0
                elif workload_raw.lower() == "teilzeit":
                    work_time = "Teilzeit"
                else:
                    # Try to parse range: "50 - 70" or "80 - 100"
                    range_match = re.search(r"(\d{1,3})\s*[--]\s*(\d{1,3})", workload_raw)
                    if range_match:
                        fte_from = float(range_match.group(1)) / 100
                        fte_to = float(range_match.group(2)) / 100
                        work_time = "Vollzeit" if fte_to >= 0.9 else "Teilzeit"
                    else:
                        # Single number: "100" or "60"
                        single_match = re.search(r"(\d{1,3})", workload_raw)
                        if single_match:
                            pct = float(single_match.group(1))
                            if pct <= 100:
                                fte_from = pct / 100
                                fte_to = pct / 100
                                work_time = "Vollzeit" if pct >= 90 else "Teilzeit"

            # --- Extract canton from URL path ---
            # URL format: /jobs/zurich/medizin-arzte-pflege/133688-...
            # The second path segment corresponds to the canton
            canton = None
            url_parts = detail_path.strip("/").split("/")
            if len(url_parts) >= 2:
                canton = url_parts[1]  # e.g. "zurich", "bern", "glarus"

            print(f"  ✓ {title[:60]:60s} | {company or 'N/A':25s} | {city or 'N/A'}")

            # Consolidate extracted data into dictionary
            job_data = {
                "title": title,
                "company": company,
                "city": city,
                "canton": canton,
                "posting_date": posting_date,
                "occupation_field": occupation_field,
                "org_type": org_type,
                "workload_raw": workload_raw,
                "fte_from": fte_from,
                "fte_to": fte_to,
                "work_time": work_time,
                "detail_url": detail_url,
            }

            jobs.append(job_data)

        except Exception as e:
            print(f"  Skipping job due to error: {e}")
            continue

    print(f"  Total so far: {len(jobs)}")

    # --- Pagination: click "Nächste Seite" button ---
    if page >= max_pages:
        break

    try:
        # Find the "Nächste Seite" (next page) button
        next_button = driver.find_element(
            By.CSS_SELECTOR, 'button[aria-label="Nächste Seite"]'
        )

        # Check if button is disabled (last page reached)
        is_disabled = next_button.get_attribute("aria-disabled")
        if is_disabled == "true":
            print("  Last page reached: stopping")
            break

        # Scroll to button so it's visible
        driver.execute_script("arguments[0].scrollIntoView(true);", next_button)
        time.sleep(0.5)

        # Click via JavaScript to avoid click-interception by overlays
        driver.execute_script("arguments[0].click();", next_button)

    except Exception as e:
        print(f"  Pagination error: {e} → stopping")
        break


print(f"\n=== Total Jobs scraped: {len(jobs)} ===")

# --- Create DataFrame and export as JSON ---
df = pd.DataFrame(jobs)

print(f"\nColumns: {list(df.columns)}")
print(f"Shape: {df.shape}")

# Export to JSON (preferred over CSV for encoding safety and null handling)
df.to_json(path_df_jobs_raw, orient="records", indent=2, force_ascii=False)
print(f"\nExported raw data to: {path_df_jobs_raw}")

driver.quit()
print("Browser closed")

2.4 Detail Page Scraping: Benefit Extraction

The overview cards on publicjobs.ch display only a subset of each listing’s metadata. The full set of fringe benefits (e.g., “Beitrag ÖV Abo”, “Home-Office”, “Gute Vorsorgeleistungen”) is only available on the individual detail pages. To answer RQ3 (Benefits & Attractiveness), we therefore implement a second scraping pass that visits each job’s detail_url and extracts the benefit tags from the detail page footer.

On each detail page, benefits are rendered inside a footer container (div.organism-business-jobs-detail__footer-benefits). Each individual benefit is a <button> element with class molecule-action-tag, and the benefit text is located in a nested <span class="molecule-typography-label__value">. The scraper extracts all benefit labels per listing and stores them as a list of strings.

To manage the volume of requests (~1’460 detail pages), the scraper uses parallel processing with multiple browser instances. Each worker runs its own headless Chrome session and processes a share of the URLs. Results are collected and saved after all workers complete their batch. A JSON checkpoint file allows the process to resume after interruption without re-scraping already visited URLs. Polite pauses between requests reduce server load per worker.

from concurrent.futures import ThreadPoolExecutor, as_completed
import threading

# --- Detail Page Scraper: Benefits (Parallel) ---
# This script visits each job's detail_url to extract benefit tags.
# It requires the overview scrape (path_df_jobs_raw) to be completed first.

# Configuration
n_workers = 5                # number of parallel browser instances
pause_between = (1.5, 3.0)   # random pause range (seconds) between requests per worker
checkpoint_every = 50        # save checkpoint every N results (across all workers)

# Thread-safe lock for writing results
lock = threading.Lock()


def create_headless_chrome():
    """Creates a headless Chrome WebDriver instance."""
    options = webdriver.ChromeOptions()
    options.add_argument("--headless=new")
    options.add_argument("--disable-blink-features=AutomationControlled")
    options.add_experimental_option("excludeSwitches", ["enable-automation"])
    return webdriver.Chrome(options=options)


def scrape_benefits_from_url(url):
    """
    Opens a detail page in a thread-local browser and extracts benefit tags.

    Args:
        url: detail page URL

    Returns:
        dict with detail_url and list of benefit strings
    """
    # Each thread gets its own browser (stored in thread-local storage)
    if not hasattr(thread_local, "driver"):
        thread_local.driver = create_headless_chrome()

    driver = thread_local.driver

    try:
        driver.get(url)
        time.sleep(random.uniform(*pause_between))

        soup = BeautifulSoup(driver.page_source, "html.parser")

        # Find the benefits container in the detail page footer
        benefits_container = soup.find(
            "div",
            class_=lambda c: c and "organism-business-jobs-detail__footer-benefits" in c
        )

        benefits = []
        if benefits_container:
            # Each benefit is a <button class="molecule-action-tag ...">
            # with text inside <span class="molecule-typography-label__value">
            benefit_buttons = benefits_container.find_all(
                "button",
                class_=lambda c: c and "molecule-action-tag" in c
            )
            for btn in benefit_buttons:
                value_span = btn.find(
                    "span", class_="molecule-typography-label__value"
                )
                if value_span:
                    text = value_span.get_text(strip=True)
                    if text:
                        benefits.append(text)

        return {"detail_url": url, "benefits": benefits}

    except Exception as e:
        print(f"\n  Failed {url}: {e}")
        return {"detail_url": url, "benefits": []}


# --- Main execution ---

# Thread-local storage for browser instances (one per worker thread)
thread_local = threading.local()

if not os.path.exists(path_df_jobs_raw):
    print("Raw data not found - run the overview scraper (section 2.3) first.")
else:
    df_overview = pd.read_json(path_df_jobs_raw)
    all_urls = df_overview["detail_url"].tolist()
    total_nr = len(all_urls)
    print(f"Total detail URLs: {total_nr}")

    # Check for existing checkpoint (resume support)
    if os.path.exists(path_df_jobs_benefits):
        df_existing = pd.read_json(path_df_jobs_benefits)
        scraped_urls = set(df_existing["detail_url"].dropna())
        remaining_urls = [u for u in all_urls if u not in scraped_urls]
        print(f"Resuming - already scraped: {len(scraped_urls)}, remaining: {len(remaining_urls)}")
    else:
        df_existing = pd.DataFrame(columns=["detail_url", "benefits"])
        remaining_urls = all_urls
        print("Starting fresh - no checkpoint found.")

    # Collect results
    all_results = list(df_existing.to_dict("records"))
    completed = len(all_results)

    print(f"Starting {n_workers} parallel workers for {len(remaining_urls)} URLs...")

    with ThreadPoolExecutor(max_workers=n_workers) as executor:
        # Submit all remaining URLs
        future_to_url = {
            executor.submit(scrape_benefits_from_url, url): url
            for url in remaining_urls
        }

        for future in as_completed(future_to_url):
            result = future.result()
            all_results.append(result)
            completed += 1

            # Progress
            pct = (completed / total_nr) * 100
            sys.stdout.write(f"\rCompleted {completed}/{total_nr} ({pct:.1f}%)")
            sys.stdout.flush()

            # Periodic checkpoint save
            if completed % checkpoint_every == 0:
                with lock:
                    pd.DataFrame(all_results).to_json(
                        path_df_jobs_benefits,
                        orient="records", indent=2, force_ascii=False
                    )

    # Final save
    df_all_benefits = pd.DataFrame(all_results)
    df_all_benefits.to_json(
        path_df_jobs_benefits,
        orient="records", indent=2, force_ascii=False
    )

    # Clean up: close all thread-local browsers
    # (ThreadPoolExecutor doesn't expose thread locals directly,
    #  but browsers will be garbage-collected when the script ends)

    print(f"\n\nDetail scraping complete: {len(df_all_benefits)}/{total_nr}")
    print(f"Saved to: {path_df_jobs_benefits}")

2.5 Integration of BFS Reference Data

To contextualize our scraped data, we enriched the dataset with official employment statistics from the Swiss Federal Statistical Office (BFS). The BESTA (Beschäftigungsstatistik) provides quarterly employment counts broken down by workload type (Vollzeit / Teilzeit) and serves as the benchmark for the RQ2 flexibility comparison.

2.5.1 BFS BESTA: Employment by Sector and Beschäftigungsgrad (via PX-Web API)

To compare our scraped workload distributions with official statistics, we obtain employment data from the BFS Beschäftigungsstatistik (BESTA). This dataset provides quarterly employment counts broken down by Wirtschaftsabteilung (economic division, NOGA), Beschäftigungsgrad (full-time vs. part-time), and Geschlecht (gender).

The data is retrieved programmatically from the BFS PX-Web API using a POST request with a JSON query body. The table px-x-0602000000_101 contains the relevant data. We request the most recent available year and all economic divisions to construct a baseline for comparison.

# --- BFS BESTA: Employment by sector & Beschäftigungsgrad ---
# Source: BFS PX-Web API, Table px-x-0602000000_101
# "Beschäftigte nach Wirtschaftsabteilung, Beschäftigungsgrad und Geschlecht"
#
# Table dimensions (in order):
#   [0] Wirtschaftsabteilung  - NOGA economic divisions
#   [1] Geschlecht            - Total / Männer / Frauen
#   [2] Beschäftigungsgrad    - Total / Vollzeit / Teilzeit
#   [3] Quartal               - e.g. 2024Q3, 2024Q4, ...

bfs_pxweb_base = "https://www.pxweb.bfs.admin.ch/api/v1/de"
besta_table = "px-x-0602000000_101/px-x-0602000000_101.px"
besta_url = f"{bfs_pxweb_base}/{besta_table}"

if os.path.exists(path_df_bfs_besta):
    df_bfs_besta = pd.read_json(path_df_bfs_besta)
    print(f"BFS BESTA data loaded from cache: {len(df_bfs_besta)} rows")
else:
    # Step 1: Fetch table metadata to inspect variable codes and values
    meta_response = requests.get(besta_url, timeout=60)
    meta_response.raise_for_status()
    meta = meta_response.json()

    # Print available variables and their codes for reference
    for var in meta["variables"]:
        print(f"\n[{var['code']}] - {var['text']}")
        for code, val in zip(var["values"][:8], var["valueTexts"][:8]):
            print(f"  {code}: {val}")
        if len(var["values"]) > 8:
            print(f"  ... ({len(var['values'])} total values)")

    # Step 2: Build query
    # - Wirtschaftsabteilung: all sectors (no filter = returns all)
    # - Geschlecht: only "Total" (we don't need gender split)
    # - Beschäftigungsgrad: explicitly request Vollzeit + Teilzeit + Total
    # - Quartal: last 4 quarters (most recent year)

    # Find variable indices by code name (safer than hard-coded positions)
    var_by_code = {v["code"]: v for v in meta["variables"]}

    query_body = {
        "query": [
            {
                "code": "Geschlecht",
                "selection": {
                    "filter": "item",
                    "values": ["TOT"]  
                }
            },
            {
                "code": "Beschäftigungsgrad",
                "selection": {
                    "filter": "item",
                    "values": ["TOT", "1", "2"]  # Total, Vollzeit, Teilzeit
                }
            },
            {
                "code": "Quartal",
                "selection": {
                    "filter": "top",
                    "values": ["4"]  # last 4 quarters
                }
            }
        ],
        "response": {
            "format": "json-stat2"
        }
    }

    # Step 3: POST query to retrieve data
    print("\nSending API request...")
    response = requests.post(besta_url, json=query_body, timeout=60)
    response.raise_for_status()
    data = response.json()

    # Step 4: Parse json-stat2 into flat DataFrame
    dims = data["dimension"]
    dim_names = list(data["id"])
    dim_labels = {d: list(dims[d]["category"]["label"].values()) for d in dim_names}

    # Build cartesian product of all dimension values
    index_tuples = list(product(*[range(len(dim_labels[d])) for d in dim_names]))

    rows = []
    for idx_tuple in index_tuples:

        multiplier = 1
        for i in reversed(range(len(dim_names))):
            flat_idx += idx_tuple[i] * multiplier
            multiplier *= len(dim_labels[dim_names[i]])

        row = {}
        for i, d in enumerate(dim_names):
            row[d] = dim_labels[d][idx_tuple[i]]
        row["value"] = data["value"][flat_idx] if flat_idx < len(data["value"]) else None
        rows.append(row)

    df_bfs_besta = pd.DataFrame(rows)

    # Save to JSON cache
    df_bfs_besta.to_json(path_df_bfs_besta, orient="records", indent=2, force_ascii=False)

4.2 RQ2: Workload Flexibility - Public Sector Jobs vs. Swiss Labour Market

For 2026Q1, 59% of advertised positions are part-time (626 of 1’056) and 41% are full-time (430 of 1’056). The official BFS BESTA reference for 2025Q1 shows the exact opposite split: part-time roles account for 41% (2’262’815 of 5’511’284) and full-time for 59% (3’248’469 of 5’511’284) of total employment in Switzerland (Figure 4). A \(\chi^2\) test of independence (H\(_0\): identical part-time / full-time distribution in postings and BFS reference; H\(_1\): distributions differ) on the raw counts rejects H\(_0\) at any conventional level (\(\chi^2 = 144.11\), \(df = 1\), \(p \approx 3.36 \times 10^{-33}\)). Because the BFS sample size alone (\(\approx 5.5\) million) makes any non-trivial difference statistically significant, we additionally compute Cohen’s \(h\) on the two part-time proportions, a sample-size-independent effect-size measure: \(h \approx 0.37\), which falls in Cohen’s small-to-medium range and confirms that the gap is not merely an artefact of statistical power. The advertised jobs in this sample are therefore not only significantly but also substantively skewed toward part-time arrangements relative to the overall workforce - a pattern most plausibly driven by healthcare, social services, and education, where part-time work is structurally more common.

5 Limitations and Future Work

Several limitations should be considered when interpreting these findings. First, the dataset represents a single cross-sectional snapshot of publicjobs.ch as of March 2026. Job posting volumes may exhibit seasonal patterns - for instance, public institutions may recruit more heavily at the start of an academic or fiscal year - and without repeated scraping at different points in time it is not possible to determine whether the observed distribution is representative of typical demand or reflects seasonal peaks. Whether such seasonality exists, and how large its effect might be, remains an open question.

Second, the platform covers the public sector exclusively. Findings do not generalize to private-sector employment, and any comparison with the broader Swiss labour market (e.g., against BFS BESTA figures) should be interpreted as a structural contrast rather than a direct equivalence.

Third, canton assignment is inferred from the URL slug of the job listing rather than from a verified location field. In cases where an employer’s registered address differs from the actual workplace, or where positions are advertised cross-canton, this approach may produce misclassifications.

Fourth, the benefit extraction relies on successfully rendering each job’s detail page. Any listings for which the detail page could not be loaded would have no benefit data, potentially biasing the benefit frequency estimates.

Fifth, the dataset relies on a single source platform. publicjobs.ch is one of several aggregators used by Swiss public-sector employers; institutions that recruit exclusively through their own portals, regional aggregators, or LinkedIn are systematically under-represented. The findings should therefore be read as reflecting the publicjobs.ch listing population rather than the entirety of the Swiss public-sector labour market.

For future work, periodic re-scraping at regular intervals would enable time-series analysis and allow seasonal effects to be quantified and controlled for. Extending the scraper to comparable platforms (e.g., jobs.ch, ostjob.ch, individual cantonal portals) would mitigate the single-platform bias and enable public-private sector comparisons. Applying NLP to the free-text job descriptions could yield richer features - such as required qualifications, language requirements, or contract duration - that are not available in the structured metadata fields.

6 Conclusion

This project demonstrates that publicly available job advertisement data, collected via automated web scraping, can yield structured insights into the Swiss public sector labour market. Across 1,056 listings from publicjobs.ch, three consistent patterns emerge. Geographic demand is heavily concentrated in Zürich, with canton-level disparities likely reflecting population size and institutional density rather than deliberate recruitment policy. The workload analysis reveals that public-sector advertisers disproportionately offer part-time positions relative to the BFS employment benchmark - a structural characteristic likely driven by healthcare and education, where flexible arrangements are common practice. Benefit profiling confirms a sector-wide emphasis on long-term employment security, flexible working hours, and commuting support, with relatively little variation across cantons or contract types.

From a methodological perspective, the project illustrates both the potential and the constraints of scraping-based data collection: the dataset is broad and timely, but its snapshot nature limits causal or longitudinal inference. Future work building on this foundation - particularly repeated scraping cycles and cross-platform comparison - would substantially increase the analytical depth achievable with this approach.

# Introduction

This project analyzes job advertisements within the Swiss public sector to examine how job requirements and working conditions vary across regions, occupations, and organisation types (cantons, municipalities, hospitals, etc.). It demonstrates how web-scraped data can be gathered, cleaned, and analysed within a complete data-science workflow to answer concrete research questions through visualisation and statistical analysis. The primary data source is publicjobs.ch, a dedicated job platform for the Swiss public sector, scraped via Selenium and BeautifulSoup. To contextualise the observed workload distribution against the broader Swiss labour market, the dataset is enriched with official employment statistics from the Swiss Federal Statistical Office (BFS), specifically the *Beschäftigungsstatistik* (BESTA) retrieved via the PX-Web API.

The analysis is guided by three research questions: **(RQ1)** How does Swiss public-sector labour demand distribute geographically and by occupational class? **(RQ2)** Are job postings statistically distinguishable from the BFS workforce benchmark in terms of full-time / part-time split? **(RQ3)** Which fringe benefits are most commonly advertised, and how do they vary between part-time and full-time roles and across cantons?

# Methods

**Data sources.** Two complementary sources are used. First, 1'056 job listings are collected from publicjobs.ch (March 2026 snapshot, 73 pages × ≈20 listings) via a Selenium-based dynamic scraper that handles JavaScript rendering, cookie banners, and pagination, with randomised polite delays. For each listing, a second scraping pass visits the detail page in parallel (thread pool) to extract advertised employee benefits. Second, the BFS BESTA reference dataset (table `px-x-0602000000_101`, "Beschäftigte nach Wirtschaftsabteilung, Beschäftigungsgrad und Geschlecht") is retrieved via the PX-Web API and parsed from JSON-stat2 format into a tidy DataFrame. Compliance with `robots.txt` was verified before scraping.

**Cleansing.** Posting dates are parsed from `DD.MM.YYYY`, FTE columns are converted to numeric and reversed ranges (e.g. `100 - 60`) are corrected, duplicates and rows with missing critical fields are dropped. Canton URL slugs are normalised to standard German canton names. Each listing is enriched with (i) a heuristic seniority class (Einstieg / Fachkraft / Projektleitung / Stellvertretung / Führung) derived from title keywords, and (ii) an ISCO-08 major group derived from the publicjobs.ch occupation field via an explicit mapping table. Because publicjobs.ch truncates long category names at 33 characters, the ISCO mapping is augmented with the truncated forms so the join is complete (1'056 / 1'056 listings classified). Benefit chips are merged in via the listing's `detail_url` and one-hot encoded.

**Analysis.** RQ1 is addressed descriptively with a canton-level choropleth, a seniority bar chart, and an ISCO bar chart. RQ2 combines an exploratory boxplot of FTE by seniority and a stacked-bar chart of part-time / full-time share by canton with a confirmatory $\chi^2$ test of independence comparing the 2026Q1 listings against the 2025Q1 BFS BESTA benchmark. RQ3 ranks the top 10 benefits by overall frequency, splits the rankings between part-time and full-time postings, and visualises geographic variation with a canton × benefit heatmap. ## Dynamic Data Extraction with Selenium

The following script initializes a Chrome WebDriver, navigates the job platform, explicitly declines tracking cookies via DOM interaction, and systematically extracts structured information from the job cards. publicjobs.ch renders its listings dynamically via JavaScript, requiring Selenium to interact with the live DOM rather than parsing static HTML alone.

The scraper performs the following steps:

1. A Chrome WebDriver instance is started and navigated to the `/jobs` overview page. After the initial page load, a cookie consent banner is dismissed via an explicit DOM click on the decline button.
2. On each page, BeautifulSoup parses the rendered HTML to locate all job card elements (`div.molecule-business-advertisement`). For each card, the scraper extracts the job title and detail URL from the first anchor tag, the company name from a labeled span element, and structured metadata fields (posting date, occupation field, workload, city, organization type) by matching Font Awesome icon classes to their corresponding value spans.
3. The raw workload string (e.g., "80 - 100", "Vollzeit", "60") is parsed into normalized FTE boundaries (`fte_from`, `fte_to`) and a categorical work-time label. The canton is inferred from the URL path structure.
4. After all cards on a page are processed, the scraper clicks the "Nächste Seite" pagination button using JavaScript execution (to avoid overlay-related click interception) and waits a randomized interval before loading the next page. A duplicate check based on detail URLs prevents repeated entries.

```{python}
    soup = BeautifulSoup(html, "html.parser")

    articles = soup.find_all("div", class_="molecule-business-advertisement")

                continue

            detail_url = base_url + detail_path

            if detail_url in seen_urls:

                "span", class_=lambda c: c and "molecule-business-advertisement__owner" in c

                    continue

                # Find the icon inside this label
                icon = label.find("i", class_=lambda c: c and "atom-icon-fontawesome" in c)

                value_span = label.find("span", class_="molecule-typography-label__value")

                if not icon or not value_span:

                icon_classes = " ".join(icon.get("class", []))
                value = value_span.get_text(strip=True)

                elif "fa-briefcase" in icon_classes:
```

## Detail Page Scraping: Benefit Extraction

The overview cards on publicjobs.ch display only a subset of each listing's metadata. The full set of fringe benefits (e.g., "Beitrag ÖV Abo", "Home-Office", "Gute Vorsorgeleistungen") is only available on the individual detail pages. To answer **RQ3 (Benefits & Attractiveness)**, we therefore implement a second scraping pass that visits each job's `detail_url` and extracts the benefit tags from the detail page footer.

On each detail page, benefits are rendered inside a footer container (`div.organism-business-jobs-detail__footer-benefits`). Each individual benefit is a `<button>` element with class `molecule-action-tag`, and the benefit text is located in a nested `<span class="molecule-typography-label__value">`. The scraper extracts all benefit labels per listing and stores them as a list of strings.

To manage the volume of requests (~1'460 detail pages), the scraper uses parallel processing with multiple browser instances. Each worker runs its own headless Chrome session and processes a share of the URLs. Results are collected and saved after all workers complete their batch. A JSON checkpoint file allows the process to resume after interruption without re-scraping already visited URLs. Polite pauses between requests reduce server load per worker.

```{python}
#| eval: false

from concurrent.futures import ThreadPoolExecutor, as_completed
import threading

# --- Detail Page Scraper: Benefits (Parallel) ---
# This script visits each job's detail_url to extract benefit tags.
# It requires the overview scrape (path_df_jobs_raw) to be completed first.

# Configuration
n_workers = 5                # number of parallel browser instances
pause_between = (1.5, 3.0)   # random pause range (seconds) between requests per worker
checkpoint_every = 50        # save checkpoint every N results (across all workers)

# Thread-safe lock for writing results
lock = threading.Lock()


def create_headless_chrome():
    """Creates a headless Chrome WebDriver instance."""
    options = webdriver.ChromeOptions()
    options.add_argument("--headless=new")
    options.add_argument("--disable-blink-features=AutomationControlled")
    options.add_experimental_option("excludeSwitches", ["enable-automation"])
    return webdriver.Chrome(options=options)


def scrape_benefits_from_url(url):
    """
    Opens a detail page in a thread-local browser and extracts benefit tags.

    Args:
        url: detail page URL

    Returns:
        dict with detail_url and list of benefit strings
    """
    # Each thread gets its own browser (stored in thread-local storage)
    if not hasattr(thread_local, "driver"):
        thread_local.driver = create_headless_chrome()

    driver = thread_local.driver

    try:
        driver.get(url)
        time.sleep(random.uniform(*pause_between))

        soup = BeautifulSoup(driver.page_source, "html.parser")

        # Find the benefits container in the detail page footer
        benefits_container = soup.find(
            "div",
            class_=lambda c: c and "organism-business-jobs-detail__footer-benefits" in c
        )

        benefits = []
        if benefits_container:
            # Each benefit is a <button class="molecule-action-tag ...">
            # with text inside <span class="molecule-typography-label__value">
            benefit_buttons = benefits_container.find_all(
                "button",
                class_=lambda c: c and "molecule-action-tag" in c
            )
            for btn in benefit_buttons:
                value_span = btn.find(
                    "span", class_="molecule-typography-label__value"
                )
                if value_span:
                    text = value_span.get_text(strip=True)
                    if text:
                        benefits.append(text)

        return {"detail_url": url, "benefits": benefits}

    except Exception as e:
        print(f"\n  Failed {url}: {e}")
        return {"detail_url": url, "benefits": []}


# --- Main execution ---

# Thread-local storage for browser instances (one per worker thread)
thread_local = threading.local()

if not os.path.exists(path_df_jobs_raw):
    print("Raw data not found - run the overview scraper (section 2.3) first.")
else:
    df_overview = pd.read_json(path_df_jobs_raw)
    all_urls = df_overview["detail_url"].tolist()
    total_nr = len(all_urls)
    print(f"Total detail URLs: {total_nr}")

    # Check for existing checkpoint (resume support)
    if os.path.exists(path_df_jobs_benefits):
        df_existing = pd.read_json(path_df_jobs_benefits)
        scraped_urls = set(df_existing["detail_url"].dropna())
        remaining_urls = [u for u in all_urls if u not in scraped_urls]
        print(f"Resuming - already scraped: {len(scraped_urls)}, remaining: {len(remaining_urls)}")
    else:
        df_existing = pd.DataFrame(columns=["detail_url", "benefits"])
        remaining_urls = all_urls
        print("Starting fresh - no checkpoint found.")

    # Collect results
    all_results = list(df_existing.to_dict("records"))
    completed = len(all_results)

    print(f"Starting {n_workers} parallel workers for {len(remaining_urls)} URLs...")

    with ThreadPoolExecutor(max_workers=n_workers) as executor:
        # Submit all remaining URLs
        future_to_url = {
            executor.submit(scrape_benefits_from_url, url): url
            for url in remaining_urls
        }

        for future in as_completed(future_to_url):
            result = future.result()
            all_results.append(result)
            completed += 1

            # Progress
            pct = (completed / total_nr) * 100
            sys.stdout.write(f"\rCompleted {completed}/{total_nr} ({pct:.1f}%)")
            sys.stdout.flush()

            # Periodic checkpoint save
            if completed % checkpoint_every == 0:
                with lock:
                    pd.DataFrame(all_results).to_json(
                        path_df_jobs_benefits,
                        orient="records", indent=2, force_ascii=False
                    )

    # Final save
    df_all_benefits = pd.DataFrame(all_results)
    df_all_benefits.to_json(
        path_df_jobs_benefits,
        orient="records", indent=2, force_ascii=False
    )

    # Clean up: close all thread-local browsers
    # (ThreadPoolExecutor doesn't expose thread locals directly,
    #  but browsers will be garbage-collected when the script ends)

    print(f"\n\nDetail scraping complete: {len(df_all_benefits)}/{total_nr}")
    print(f"Saved to: {path_df_jobs_benefits}")
```

```{python}
def parse_employment(value):
    value = str(value).strip()
```

## RQ2: Workload Flexibility - Public Sector Jobs vs. Swiss Labour Market

For 2026Q1, **59% of advertised positions are part-time (626 of 1'056)** and **41% are full-time (430 of 1'056)**. The official BFS BESTA reference for 2025Q1 shows the exact opposite split: part-time roles account for 41% (2'262'815 of 5'511'284) and full-time for 59% (3'248'469 of 5'511'284) of total employment in Switzerland (@fig-flex-comparison). A $\chi^2$ test of independence (H$_0$: identical part-time / full-time distribution in postings and BFS reference; H$_1$: distributions differ) on the raw counts rejects H$_0$ at any conventional level (**$\chi^2 = 144.11$, $df = 1$, $p \approx 3.36 \times 10^{-33}$**). Because the BFS sample size alone ($\approx 5.5$ million) makes any non-trivial difference statistically significant, we additionally compute Cohen's $h$ on the two part-time proportions, a sample-size-independent effect-size measure: $h \approx 0.37$, which falls in Cohen's **small-to-medium** range and confirms that the gap is not merely an artefact of statistical power. The advertised jobs in this sample are therefore not only significantly but also substantively skewed toward part-time arrangements relative to the overall workforce - a pattern most plausibly driven by healthcare, social services, and education, where part-time work is structurally more common.

## RQ3: Benefits and Attractiveness

The benefit analysis (@fig-benefits-frequency) shows that Swiss public-sector employers predominantly advertise non-monetary fringe benefits. **Mitarbeitendenförderung** (staff development support, 48% of listings) and **Gute Vorsorgeleistungen** (strong pension provisions, 46%) are the most frequently offered benefits, reflecting the sector's traditionally strong focus on employment security and long-term stability. **Gleitzeit** (flexible working hours, 37%) and **ÖV-Anbindung** (public-transport accessibility, 35%) also feature prominently, consistent with the predominantly urban location of the listed positions.

The distribution of these benefits is uneven between part-time and full-time job offerings. In general, benefits are mentioned less frequently in part-time postings than in full-time ones. The gap is particularly pronounced for **Mitarbeitendenförderung** (66% full-time vs. 36% part-time), **Bezahlte Pausen** (paid breaks, 48% vs. 20%), and **Beitrag Aus- und Weiterbildung** (training contributions, 46% vs. 23%) - effectively a factor of roughly two. The notable exception is **Gute Vorsorgeleistungen**, which is offered at a nearly identical rate in both workload types (45% vs. 46%), in line with Swiss occupational-pension regulations that apply irrespective of contract size. Geographically (@fig-benefits-by-canton), **Zürich** clearly provides the widest range of employee-support benefits in absolute terms - unsurprising given it hosts more than half of all listings - followed by a second tier of **Bern**, **Luzern**, and **Aargau**. The canton-level benefit profile largely mirrors listing volume rather than indicating structurally different benefit policies across regions. In summary, the availability of fringe benefits is not evenly distributed across cantons or contract types: full-time positions tend to advertise roughly twice as many top-tier benefits as part-time roles, with the important exception of pension provisions, which remain consistent across contract types and reinforce the sector's identity as a stable, long-term employer.

# Limitations and Future Work

Several limitations should be considered when interpreting these findings. First, the dataset represents a single cross-sectional snapshot of publicjobs.ch as of March 2026. Job posting volumes may exhibit seasonal patterns - for instance, public institutions may recruit more heavily at the start of an academic or fiscal year - and without repeated scraping at different points in time it is not possible to determine whether the observed distribution is representative of typical demand or reflects seasonal peaks. Whether such seasonality exists, and how large its effect might be, remains an open question.

Second, the platform covers the public sector exclusively. Findings do not generalize to private-sector employment, and any comparison with the broader Swiss labour market (e.g., against BFS BESTA figures) should be interpreted as a structural contrast rather than a direct equivalence.

Third, canton assignment is inferred from the URL slug of the job listing rather than from a verified location field. In cases where an employer's registered address differs from the actual workplace, or where positions are advertised cross-canton, this approach may produce misclassifications.

Fourth, the benefit extraction relies on successfully rendering each job's detail page. Any listings for which the detail page could not be loaded would have no benefit data, potentially biasing the benefit frequency estimates.

Fifth, the dataset relies on a single source platform. publicjobs.ch is one of several aggregators used by Swiss public-sector employers; institutions that recruit exclusively through their own portals, regional aggregators, or LinkedIn are systematically under-represented. The findings should therefore be read as reflecting the publicjobs.ch listing population rather than the entirety of the Swiss public-sector labour market.

For future work, periodic re-scraping at regular intervals would enable time-series analysis and allow seasonal effects to be quantified and controlled for. Extending the scraper to comparable platforms (e.g., jobs.ch, ostjob.ch, individual cantonal portals) would mitigate the single-platform bias and enable public-private sector comparisons. Applying NLP to the free-text job descriptions could yield richer features - such as required qualifications, language requirements, or contract duration - that are not available in the structured metadata fields.

# Conclusion

This project demonstrates that publicly available job advertisement data, collected via automated web scraping, can yield structured insights into the Swiss public sector labour market. Across 1,056 listings from publicjobs.ch, three consistent patterns emerge. Geographic demand is heavily concentrated in Zürich, with canton-level disparities likely reflecting population size and institutional density rather than deliberate recruitment policy. The workload analysis reveals that public-sector advertisers disproportionately offer part-time positions relative to the BFS employment benchmark - a structural characteristic likely driven by healthcare and education, where flexible arrangements are common practice. Benefit profiling confirms a sector-wide emphasis on long-term employment security, flexible working hours, and commuting support, with relatively little variation across cantons or contract types.

From a methodological perspective, the project illustrates both the potential and the constraints of scraping-based data collection: the dataset is broad and timely, but its snapshot nature limits causal or longitudinal inference. Future work building on this foundation - particularly repeated scraping cycles and cross-platform comparison - would substantially increase the analytical depth achievable with this approach.
